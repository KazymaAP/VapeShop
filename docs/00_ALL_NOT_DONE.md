## 🔴 КРИТИЧЕСКИЕ ПРОБЛЕМЫ (Production-Blocking)

### Безопасность

#### 1. ⚠️ КРИТИЧНО: X-Telegram-Id не криптографически защищена

- **Файл:** `lib/auth.ts:16-21`
- **Проблема:** Заголовок `X-Telegram-Id` передается в открытом виде и может быть подделан. Функция `verifyTelegramInitData()` существует, но не применяется во всех API эндпоинтах.
- **Почему опасно:** Злоумышленник может подделать любой Telegram ID и получить доступ к чужим заказам, данным, админ-панели.
- **Где это видно:** `pages/admin/index.tsx:80` отправляет неверифицированный запрос, `pages/api/admin/orders.ts:34` логирует действие без проверки.
- **Исправить:**
  - Применить HMAC-SHA256 верификацию ко всем приходящим заголовкам
  - Использовать `verifyTelegramInitData()` в критичных местах
  - Или передавать инициализированный `initData` вместо простого ID
- **Время:** 6 часов
- **Статус:** не начато

#### 2. ⚠️ КРИТИЧНО: /api/orders.ts не защищен requireAuth

- **Файл:** `pages/api/orders.ts:9`
- **Проблема:** API создания заказов НЕ использует `requireAuth()`. Может быть вызван без аутентификации или с поддельным Telegram ID.
- **Код проблемы:**
  ```typescript
  async function handler(req: NextApiRequest, res: NextApiResponse) {
    // Нет проверки аутентификации!
    const currentTelegramId = await getTelegramIdFromRequest(req);
  ```
- **Риск:** Спам-атаки на создание заказов, создание заказов от других пользователей, DoS.
- **Исправить:**
  ```typescript
  export default requireAuth(handler, ['customer', 'buyer']);
  ```
- **Время:** 1 час
- **Статус:** не начато

#### 3. ⚠️ КРИТИЧНО: Большинство admin API не защищены

- **Файл:** `pages/api/admin/*`
- **Проблема:** Из 50+ API эндпоинтов защищено только несколько. Не защищены:
  - `/api/admin/users.ts` — может менять роли для всех пользователей
  - `/api/admin/stats.ts` — может вытащить все статистику
  - `/api/admin/settings.ts` — может менять глобальные настройки
  - `/api/admin/broadcast.ts` — может отправлять сообщения от бота
  - `/api/admin/faq.ts` — может менять FAQ
  - `/api/admin/banners.ts` — может менять баннеры
  - `/api/admin/import.ts` — может загружать CSV файлы
  - `/api/admin/alerts.ts` — может менять уведомления
  - `/api/admin/roles.ts` — может менять роли
  - и еще 10+ других
- **Где проверить:** Запроси файл и посмотри на строку `export default ...`. Если там не `requireAuth()`, то не защищен.
- **Исправить:** Обернуть все в `requireAuth(handler, ['admin'])` или `requireAuth(handler, ['admin', 'manager'])`
- **Время:** 12 часов (для всех эндпоинтов)
- **Статус:** частично сделано (1 из 15+)

#### 4. ⚠️ КРИТИЧНО: Нет проверки блокировки на платежах

- **Файл:** `pages/api/orders.ts:80`, `lib/bot/payments.ts`
- **Проблема:** Если пользователь заблокирован (`users.is_blocked = true`), он может все еще создавать заказы и платить через Stars.
- **Исправить:** Добавить проверку в начало handler:
  ```typescript
  const isBlocked = await isUserBlocked(telegramId);
  if (isBlocked) return res.status(403).json({ error: 'User is blocked' });
  ```
- **Время:** 2 часа
- **Статус:** не начато

#### 5. ⚠️ КРИТИЧНО: SQL injection risk в некоторых местах

- **Файл:** `pages/api/products/filters.ts:45`, `pages/api/admin/kanban.ts`
- **Проблема:** Некоторые динамические параметры в WHERE clause могут содержать неверифицированные значения.
- **Пример:** Если `status` параметр используется напрямую без проверки, может быть проблема.
- **Исправить:** Всегда использовать параметризованные запросы `$1, $2` вместо string interpolation.
- **Время:** 3 часа
- **Статус:** не начато

#### 6. ⚠️ КРИТИЧНО: Rate limiting не применяется везде

- **Файл:** `lib/rateLimit.ts` существует, но не используется в большинстве API
- **Проблема:** На критичные эндпоинты (платежи, создание заказов, изменение ролей) не применен rate limiting.
- **Исправить:**
  ```typescript
  async function handler(req, res) {
    const rateLimitResult = await rateLimit(req, 'create_order', 10); // макс 10 в час
    if (!rateLimitResult.success) return res.status(429).json({ error: 'Rate limit exceeded' });
  ```
- **Время:** 4 часа
- **Статус:** не начано

#### 7. ⚠️ КРИТИЧНО: Telegram bot webhook не верифицируется

- **Файл:** `pages/api/bot.ts:1`
- **Проблема:** Webhook не проверяет `X-Telegram-Bot-Api-Secret-Token`. Может быть использован для отправки поддельных update'ов.
- **Исправить:** Добавить проверку:
  ```typescript
  const token = req.headers['x-telegram-bot-api-secret-token'];
  if (token !== process.env.TELEGRAM_BOT_SECRET) {
    return res.status(401).json({ error: 'Unauthorized' });
  }
  ```
- **Примечание:** Нужно установить `TELEGRAM_BOT_SECRET` в `.env.local` и у BotFather
- **Время:** 2 часа
- **Статус:** не начато

#### 8. ⚠️ КРИТИЧНО: Нет CSRF защиты

- **Файл:** Везде в POST/PUT/DELETE запросах
- **Проблема:** Нет CSRF token валидации. Можно сделать действие от имени пользователя с другого сайта.
- **Исправить:** Или использовать SameSite=Strict cookies (уже установлено для session), или добавить CSRF token.
- **Время:** 3 часа
- **Статус:** не начато
