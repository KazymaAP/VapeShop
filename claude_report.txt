[CRIT-002] CSRF защита не работает в production
Файл: lib/csrf.ts, строки 20, 62-95
Проблема: csrfStore хранится в памяти процесса Node.js. На Vercel каждый serverless
function вызов — новый процесс. csrfStore будет пустым при каждом запросе, CSRF
токены не найдутся, verifyCSRFToken() всегда вернёт false. Вся CSRF защита
фактически отключена, но при этом блокирует запросы пользователей.
Как есть:
  const csrfStore: CSRFStore = {};  // in-memory, не работает на Vercel
Исправление: Использовать Redis (Upstash) или убрать CSRF для Telegram Mini App
(там уже есть HMAC-валидация initData). Если оставлять — хранить токены в Redis:
  import { Redis } from '@upstash/redis';
  const redis = new Redis({ url: process.env.UPSTASH_URL, token: process.env.UPSTASH_TOKEN });
  await redis.set(`csrf:${sessionId}`, token, { ex: 86400 });
Приоритет: 🔴 Критический

[CRIT-003] Rate limiting не работает в production
Файл: lib/rateLimit.ts, строки 7-10
Проблема: const store: RateLimitStore = {} — in-memory хранилище. Та же проблема
что с CSRF: на Vercel serverless каждый запрос может идти в новый процесс. Rate
limiting полностью нефункционален — злоумышленник может делать неограниченное
количество запросов.
Исправление: Перенести в Redis (Upstash) с использованием sliding window алгоритма.
Минимально рабочее решение — использовать заголовки X-RateLimit-* от Vercel Edge,
или подключить @upstash/ratelimit:
  import { Ratelimit } from "@upstash/ratelimit";
  const ratelimit = new Ratelimit({ redis, limiter: Ratelimit.slidingWindow(10, "1 m") });
Приоритет: 🔴 Критический

[CRIT-006] Race condition при создании заказа
Файл: pages/api/orders.ts, строки ~65-120
Проблема: Проверка остатков товаров и создание заказа выполняются в транзакции,
но проверка SELECT ... FOR UPDATE не используется. При параллельных запросах оба
могут пройти проверку наличия, и товар будет продан дважды.
Как есть:
  const productsRes = await query('SELECT id, stock, price FROM products WHERE...');
  // ... проверка остатков ...
  await transaction(async (client) => { ... UPDATE products SET stock = stock - $1 ... });
Исправление: Добавить FOR UPDATE в SELECT внутри транзакции:
  const productsRes = await client.query(
    'SELECT id, stock, price FROM products WHERE id = ANY($1) FOR UPDATE', [productIds]
  );

Проблема: Запрос:
  WHERE LOWER(name) ILIKE $1 OR LOWER(specification) ILIKE $1
LOWER() делает индекс idx_products_name неэффективным. Нужен функциональный индекс.
Исправление: Использовать ILIKE напрямую (PostgreSQL case-insensitive через pg_trgm):
  CREATE EXTENSION IF NOT EXISTS pg_trgm;
  CREATE INDEX idx_products_name_trgm ON products USING gin(name gin_trgm_ops);
Приоритет: 🟡 Средний

ИТОГОВОЕ ЗАКЛЮЧЕНИЕ
───────────────────────────────────────────────────────────────────────────────

Проект VapeShop имеет хорошую архитектурную основу и широкий функционал. Основные
проблемы можно разделить на три категории:

1. КРИТИЧЕСКИЕ ДЕФЕКТЫ (9 проблем): Несколько файлов не будут компилироваться
   из-за отсутствующих импортов (rbac.ts, pages.ts). Утечка секретов. Race condition
   в платёжной системе. In-memory хранилища которые не работают на Vercel.

2. НЕСООТВЕТСТВИЕ СХЕМЫ БД (20+ проблем): Код и миграции используют разные имена
   таблиц и полей. Множество таблиц создаётся несколько раз в разных миграциях
   с разными схемами. Это главная архитектурная проблема.

3. ЛОГИЧЕСКИЕ ОШИБКИ В ЗАПРОСАХ (15+ проблем): Большинство API endpoint'ов к
   users, orders используют users.id вместо users.telegram_id. Эти endpoint'ы
   вернут пустые результаты или ошибки.

После устранения всех критических и высокоприоритетных проблем (3-4 недели работы
опытного разработчика) проект будет готов к production деплою на Vercel с Neon БД.
